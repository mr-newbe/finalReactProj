<div>로컬에서 사용하기에는 충분한듯 하고, 후에 충분히 시간을 들여 리펙토링 등을 하도록 하겠습니다</div>

<div>두번째 페이지이자 메인 페이지입니다. 이곳에서 추가 버튼으로 할일을 추가하고 알림 버튼으로 가운데 영역으로 옮깁니다.</div>
<div>두번째 페이지에서 드래그를 통해 몇시에 그 일을 할지를 정합니다. </div>
<div>그리고 세번째 달력에서 해당 달에 대해서만 정할 수 있는데, 더블클릭으로 그달의 일정을 추가하면 로컬 스토리지에 데이터가 저장, 후에 불러와집니다.</div>
<div>가운데 창 가장 밑의 버튼을 누르고 다음 페이지로 넘거갑니다.</div>
<img width="1280" alt="image" src="https://github.com/mr-newbe/finalReactProj/assets/49405838/f3ba58bd-d85f-471f-8897-7f07a3b5d7cd">
<div>모든 요소를 버튼으로 설정했습니다</div>
<div>버튼을 누르면 그 부분이 초록으로 바뀌고, 그 뜻은 해당 일이 완료되었다는 뜻입니다. 마찬가지로 제일 아래 버튼을 통해 다음 페이지로 이동합니다</div>
<img width="1280" alt="image" src="https://github.com/mr-newbe/finalReactProj/assets/49405838/d68a0468-353a-4f0c-bbb9-c8adaf113512">
<div>마지막 페이지입니다. 마지막 버튼을 누르면 처음 페이지로 돌아가나 그게 중요한게 아니죠</div>
<div>이곳에 있는 일기 자동 생성 버튼, 이것이 이 프로젝트의 핵심입니다.</div>
<div>오늘의 날씨, 기온, 하루 일정의 마무리 정도, 앞으로 있을 이번달의 일정을 바탕으로 "사실에 기반한 일기를 작성해줍니다"</div>
<div>바로 밑의 quill Editor로 그 만들어진 일기가 입력됩니다</div>
<div>일기에 오늘의 감정적 사건들, 비하인드 스토리들을 작성하고 나면 밑의 인증 버튼과 만든일기 게시 버튼을 눌러 blogger api를 통해 블로그에 게시합니다</div>
<div>그 일기의 제목은 그날의 날짜(오전 4시 기준)를 바탕으로 바뀌며 일기는 설정한 페이지로 이동합니다</div>
<br/>
<br/>
<br/>
<div>기능들을 설명했으니 이번엔 기술에 대해 이야기해보겠습니다</div>
<div>우선 페이지 이동에 쓰인 기술은 간단합니다. display block과 none으로 페이지가 이동하는듯하게 꾸몄습니다.</div>
<div>그 이유는 github page에서는 react의 react router가 적용되지 않기 때문입니다. </div>
<div>첫번째 페이지에서는 타이머, 기상청 api, 그리고 랜덤함수로 바뀌는 명언이 있습니다.</div>
<div>전체적으로 recoil이 쓰였는데, 이는 전역변수를 통해 props를 덜 사용하기 위함이였습니다. </div>
<div>axios를 사용해 api를 끌어왔으며 promise의 일부인 then을 통해 비동기에 대한 처리를 하였습니다.</div>
<div>api의 응답 형식은 대부분 json이였고, SJON.parse를 통해 데이터를 추출할 수 있었습니다.</div>
<div>조금 위험하지만 가장 효과적이였던 방법으로 innerHTML을 자주 사용했고 기상청에서 사용되는 xy좌표를</div>
<div>구하기 위한 외부 코드도 복사해서 넣었고, 400줄가량의 분량이 나왔습니다.</div>
<div>두번째 페이지에는 대부분의 기능이 모여있는 만큼 굉장히 많은 기술이 들어갔습니다</div>
<div>Sortablejs라는 외부 라이브러리를 활용해 드래그가 가능하도록 2페이지의 가운데 요소를 만들었고</div>
<div>styled component를 같은 동일 페이지에 정의함으로서 간단한 css의 적용을 완료했습니다.</div>
<div>컴포넌트를 잘 구분하고 나누고 삽입함으로서 구조적인 코드구성을 세우려 했지만,.... 너무 들어가는 기능이</div>
<div>많아지다보니 여러 문제가 발생했다고 생각합니다. 그 대표격이 바로 다량의 전역변수이죠</div>
<div>달력 또한 외부의 코드를 가져왔습니다. 하지만 저의 프로젝트와 여러모로 디자인이 맞지 않았고, 따라서</div>
<div>여러번의 수정으로 기념일 적용을 제외한 나머지 기능들의 구현에 성공했습니다.</div>
<div>드래그한 요소들의 적용은 그 즉시 document,getEle... 를 통해 배치된 todo의 순서를 파악, 배열로서 저장,</div>
<div>2.5페이지는 긴급하게 만듬으로서 디자인이 별로이고, 정말 필요성만 강조되었는데 버튼으로 색을 변경하고</div>
<div>그 색을 판별하는 것으로 할일의 완료 여부를 판단했습니다</div>
<div>마지막 페이지에서는 Chat.js 파일에서 그 진가가 들어나는데, 여간의 데이터를 배열 형태로</div>
<div>전역변수에 저장했습니다. 또한 quill Editor를 통해 chat gpt의 제페토 모델로 받은 일기 글을 </div>
<div>받아주었습니다. 하지만 quill 에서 사진을 넣어려 할 때 문제점이 있는데, 사진의 실제 형태가 16진수로</div>
<div>한 30줄 넘게 만들어져 있기에 데이터 형태로서 문제가 많습니다.</div>
<div>따라서 firebase를 썼고, 그 정보를 firebase에 임시로 게시함으로서 사진의 주소를 획기적으로 줄였습니다.</div>
<div>마지막으론 blogger api, 구글의 인증받식을 채택하고 있기에 (Oauth) 구글 로그인과 그 권한 양도를 통해</div>
<div>서야 비로소 기blogger api를 사용할 수 있었습니다.</div>
